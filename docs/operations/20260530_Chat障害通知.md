## 1. 概要

Cloud Run `pay-collector` の各エンドポイントで障害が発生した際、Google Chat スペースへ
Incoming Webhook 経由でテクニカルな障害内容を自動投稿する。

| 項目 | 内容 |
|------|------|
| 投稿方式 | Google Chat Incoming Webhook（一方向通知） |
| webhook URL | Secret Manager `chat-webhook-url` → Cloud Run env `CHAT_WEBHOOK_URL` で注入 |
| 未設定時 | **no-op**（投稿せずバッチは正常動作）。URL 設定後に通知が有効化される |
| 通知失敗時 | バッチ本体は落とさない（送信エラーはログのみ） |
| 通知内容 | テクニカルのみ: Step名 / 例外型 / メッセージ / 時刻(JST) / リビジョン |

実装: `cloud-run/chat_notifier.py` / `cloud-run/main.py`

## 2. 通知範囲

| エンドポイント | 致命的エラー(500) | 部分失敗(集約) |
|---|---|---|
| `POST /`（毎朝バッチ） | ✅ 即通知 | ✅ Step0-7 の失敗を末尾で集約1通知 |
| `POST /update-groups` | ✅ 即通知 | — |
| `POST /sync/main-reports` | ✅ 即通知 | — |
| `POST /sync/reimbursement` | ✅ 即通知 | — |
| `POST /sync/member-master` | ✅ 即通知 | — |

毎朝バッチの部分失敗対象 Step: Step0 snapshot（テーブル単位の失敗）/ Step4 グループ情報 /
Step5 dashboard_users同期 / Step6 立替金 / Step7 タダメンM。
これらは「失敗しても本体は成功扱い」だが、障害把握のため集約通知する。

## 3. 通知メッセージの見方

### 致命的エラー（処理全体が停止、HTTP 500）

```
🔴 pay-collector 致命的エラー
発生: 2026-05-30 06:03 JST / rev: pay-collector-00030-zlc / 毎朝バッチ POST /
• HttpError: <APIエラー詳細>
```

→ バッチ全体が停止。即対応が必要。Cloud Logging で `exc_info` の完全な traceback を確認。

### 部分失敗（一部 Step のスキップ、HTTP 200）

```
🔴 pay-collector 障害 (2件)
発生: 2026-05-30 06:03 JST / rev: pay-collector-00030-zlc / 毎朝バッチ POST /
• [Step6 立替金] HttpError: The caller does not have permission
• [Step0 snapshot] check_logs の作成に失敗
```

→ 本体バッチは完了しているが、該当 Step のデータが当日更新されていない。
当該テーブル・Step を個別に確認。

> メッセージ本文（例外メッセージ）にはスプレッドシート URL やメールが含まれることがある。
> 投稿先スペースの閲覧者は運用管理者に限定すること。

## 4. 有効化手順（webhook URL 取得後）

通知は no-op 版がデプロイ済みの状態から、以下で有効化する。

### 4.1 Chat スペースで webhook URL を生成（スペース管理者）

1. 対象スペースを開く → スペース名 →「アプリと統合」
2. 「Webhook」→「追加」→ 名前（例: `pay-collector障害通知`）
3. 生成された `https://chat.googleapis.com/v1/spaces/.../messages?key=...&token=...` をコピー

### 4.2 Secret Manager に登録（N5）

```bash
# webhook URL（key/token を含む機密）を shell history に残さないよう read -s で対話入力する。
# 実行後にプロンプトへ URL を貼り付けて Enter。
read -rs CHAT_WEBHOOK_URL
printf '%s' "$CHAT_WEBHOOK_URL" \
  | gcloud secrets create chat-webhook-url --data-file=- --project=monthly-pay-tax
unset CHAT_WEBHOOK_URL  # 環境変数も残さない

# pay-collector@ SA に閲覧権限を付与（未付与の場合）
gcloud secrets add-iam-policy-binding chat-webhook-url \
  --member="serviceAccount:pay-collector@monthly-pay-tax.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor" --project=monthly-pay-tax
```

> webhook URL は `key=`/`token=` を含む機密情報。コマンドライン引数に直接書くと
> shell history に残るため、必ず上記の `read -s` 方式か Secret Manager UI から登録すること。

### 4.3 Cloud Run に env を注入（N4）

`.github/workflows/deploy-collector.yml` の `gcloud run deploy` に以下を追加してデプロイ:

```
--update-secrets=CHAT_WEBHOOK_URL=chat-webhook-url:latest
```

または手動デプロイ時に同オプションを付与。デプロイ後、`CHAT_WEBHOOK_URL` が設定され通知が有効化される。

### 4.4 動作確認

`POST /sync/member-master` 等で意図的に失敗を起こすか、次回バッチの障害発生時に
スペースへ投稿されることを確認する。

## 5. 無効化・URL 再生成

| 操作 | 方法 |
|------|------|
| 一時的に通知を止める | Cloud Run の env から `CHAT_WEBHOOK_URL` を外して再デプロイ（no-op に戻る） |
| webhook URL 漏洩・再生成 | Chat スペースで webhook を削除→再作成し、新 URL で `gcloud secrets versions add chat-webhook-url --data-file=-` |
