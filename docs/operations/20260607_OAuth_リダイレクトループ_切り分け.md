## 1. 概要

ダッシュボード (`pay-dashboard`) で「Google ログインしてもダッシュボードに到達できず、リダイレクトが繰り返される」「アクセス拒否される」等の症状が出た場合の切り分け手順。

最も典型的な原因と対策を含む既知パターンを整備済。**最初に検索: 既知症状なら 1 ステップで解消できる場合あり**。

## 2. 症状の言語化（ユーザーから聞き取り）

| 質問項目 | 確認内容 |
|---|---|
| 何度ログインしてもループする？ | (a) ループする → OAuth セッション問題 / (b) アクセス拒否画面が出る → ロール問題 |
| ブラウザは何？ | Chrome / Safari / Firefox / Edge |
| Google マルチアカウントログイン中？ | tadakayo.jp 以外のアカウントが選ばれている可能性 |
| シークレットウィンドウで再現？ | する → サーバー側 / しない → クライアント Cookie / 拡張機能干渉 |
| 「リダイレクト」が起きるのは具体的にどの画面？ | (a) `pay-dashboard...` を開いた瞬間 Google ログイン画面に飛ぶ / (b) ログイン後にアクセス拒否 / (c) ログイン後にループ |

## 3. 切り分けフローチャート

```
症状報告
├─ アクセス拒否画面が出る (role=None) → §3.1 dashboard_users 登録確認
├─ ログイン後にループ (302 連続)  → §3.2 Cloud Run ログ調査 → sessionAffinity 確認
└─ Google ログイン画面に戻される  → §3.3 クライアント環境確認
```

## 3.1 dashboard_users 登録確認

```bash
bq query --use_legacy_sql=false --format=prettyjson \
  "SELECT email, role, display_name, source_group, created_at, updated_at
   FROM \`monthly-pay-tax.pay_reports.dashboard_users\`
   WHERE LOWER(email) = LOWER('対象メールアドレス')
   LIMIT 1"
```

**結果別の対処**:

| 結果 | 対処 |
|---|---|
| 該当行なし | ユーザー管理ページで該当メールを `user` / `viewer` / `checker` / `admin` のいずれかで登録 |
| `role` が NULL | 上記同様、適切なロールを設定 |
| `role` が `user`/`viewer`/`checker`/`admin` のいずれか | ⏭️ §3.2 へ (登録 OK、別原因) |

## 3.2 Cloud Run ログで OAuth callback 頻発確認

```bash
gcloud logging read \
  'resource.type=cloud_run_revision
   AND resource.labels.service_name=pay-dashboard
   AND httpRequest.requestUrl=~".oauth2callback"' \
  --limit=30 \
  --project=monthly-pay-tax \
  --format="value(timestamp,httpRequest.status,httpRequest.requestUrl)" \
  --freshness=2h
```

### 注目パターン

- ✅ 正常: `/oauth2callback` が 1 ユーザーあたり 1〜2 回程度
- ❌ 異常: **同じ `state` パラメータが数秒間隔で複数回出現** → OAuth セッションが確定できず再認証ループ
- ❌ 異常: `prompt=none` (silent re-auth) が連続発生 → セッション切れ自動再認証の繰り返し

### 異常パターン検出時の対処

**sessionAffinity 設定確認**:

```bash
gcloud run services describe pay-dashboard \
  --region=asia-northeast1 --project=monthly-pay-tax --format=json | \
  python3 -c "import json,sys; d=json.load(sys.stdin); \
print(d['spec']['template']['metadata']['annotations'].get('run.googleapis.com/sessionAffinity', 'NOT SET'))"
```

| 結果 | 対処 |
|---|---|
| `true` | sessionAffinity は有効、別原因の可能性 (OOM / Cookie 等) → §3.4 |
| `NOT SET` / `false` | **本番影響のある修正アクション要 (下記参照)** |

**sessionAffinity 有効化**:

```bash
gcloud run services update pay-dashboard \
  --session-affinity \
  --region=asia-northeast1 \
  --project=monthly-pay-tax
```

- 副作用なし、料金影響なし
- 新リビジョンがすぐに 100% トラフィックを受信
- ロールバックは `--no-session-affinity` で即座可能
- ADR-0007 で採用根拠を記録

## 3.3 クライアント環境確認

サーバー側に問題が見つからない場合は、ユーザー側のブラウザ環境を確認:

1. **複数 Google アカウントログイン中**:
   - 個人アカウントが選択されており、tadakayo.jp ドメイン外でブロックされる
   - 対処: シークレットウィンドウで `tadakayo.jp` アカウントのみでログイン
2. **Cookie / セッション破損**:
   - 対処: ブラウザのキャッシュ + Cookie をクリア後に再アクセス
3. **Safari ITP / Third-party Cookie ブロック**:
   - 対処: Chrome / Firefox で試す
4. **ブラウザ拡張干渉** (広告ブロッカー等):
   - 対処: シークレットウィンドウで再アクセス

## 3.4 OOM (Memory) 確認

Cloud Run pay-dashboard は **Memory 512 MiB** が割り当てられている (CLAUDE.md 参照)。
OOM が頻発するとコンテナが再起動され、Streamlit セッションが失われて再認証ループの原因になる場合がある。

```bash
gcloud logging read \
  'resource.type=cloud_run_revision
   AND resource.labels.service_name=pay-dashboard
   AND severity=ERROR
   AND textPayload:"Memory limit"' \
  --limit=20 --project=monthly-pay-tax \
  --format="value(timestamp)" --freshness=7d
```

直近 7 日で **5 件以上の OOM が発生**している場合は memory 増強 (512 → 1 GiB) を検討。`--memory` フラグの変更は ADR 経由で記録すること (Cloud Run 設定変更は ADR-0004 / ADR-0007 と同様に影響範囲を整理してから実行)。

## 4. よくある誤診

| 誤った診断 | 実際の原因 |
|---|---|
| 「role が viewer なのが原因」 | viewer は正規ロール (`lib/auth.py:require_user` で許可)、認証ループとは無関係 |
| 「BQ クエリが case-sensitive で email がマッチしない」 | Google OAuth から取得した `st.user.email` は lowercase、dashboard_users 登録も通常 lowercase、case mismatch は通常起きない (`LOWER()` で更に安全側) |
| 「Cookie が短時間で expire している」 | `cookie_secret` は固定なので cookie 自体は維持される。問題は **サーバー側 state がインスタンス間で共有されない** 点 |
