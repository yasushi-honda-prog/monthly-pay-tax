## 🔍 技術調査で判明したこと（Phase 0 の一次評価）

先に結論の根拠を共有します。詳細は `REF_20260411_reimbursement-sheet-discovery.md` に記載しています。

### 発見1: 立替金シートは **既に202件配布済み** で構造も完全統一

- ゆりさんの配布ツール（`管理用_立替金等入力シート配布ツール`）が既に全メンバーに配布完了
- **ヘッダー構造は12列で全員完全一致**（年/月日/対象PJ/分類/支払用途/金額/仮払金額/区間/訪問目的/PDF保存先URL）
- タブ名も `入力シート` で統一
- **既存データ構造の変更ゼロ** でBigQuery取り込みが可能

### 発見2: 「対象PJ」プルダウンで WAM/非WAM の自動判別が可能

- 現状の値: `ケアプーPJ` / `神奈川県PJ` / `経産省PJ` / `その他`
- ここに `WAM-出張タダスクPJ`（仮称）等のWAM専用値を追加するだけで、**入力側の運用変更は最小限**（1回のプルダウン値追加のみ）
- **集計は自動** → WAM/非WAM が機械的に分離される

### 発見3: 業務報告シート側も既存 `gyomu_reports` で十分

- 既存の190件巡回インフラ（pay-collector）がそのまま使える
- 業務報告シートの **「活動分類」プルダウン** に WAM値を追加するだけで対応可能（Phase 1a）
- `v_monthly_compensation` VIEW がそのまま流用でき、要件③の基盤に

### 発見4: 既存 `hojo_reports`（月次集計）は**そのまま維持**

- WAM対応は **新規の明細レベルテーブル `reimbursement_items`** を追加する方式を推奨
- 既存の分析系（ダッシュボードの4タブ）は**一切触らない** → ブラスト半径を最小化

## 💬 相談事項①〜④への技術回答

### 相談①: 2シートのデータ統合・PDF領収書生成の実現可否と推奨アプローチ

**結論**: **実現可能、ただし既存PDFの集約を推奨**（新規生成よりコスト低）

**根拠**:
- 立替金シートには既に各トリップ単位で **「請求書のPDF保存先URL」列** が存在（ほぼ全明細に紐付け済み）
- したがって、要件①は **新規PDF生成** ではなく、**既存PDFの集約（メタデータ付き合算レシート化）** で対応可能
- 2シート統合の方式: 業務報告（業務委託費）+ 立替金（旅費・実費）を `member_nickname × 月` で JOIN して集計

**推奨アプローチ**:
```
Phase 1b:
  立替金シート(202件) → reimbursement_items (BQ)
  ↓
Phase 2 (要件①):
  v_wam_reimbursement_monthly (WAM対象のみ)
  ↓ 月別・メンバー別に集約
  既存PDFのDriveメタデータ（URL/ファイル名）を PDF1枚にまとめる
  + 業務委託費の内訳を付記
  → 1メンバー・1月・1PDF → 1振込 の実現
```

**コスト感**: 既存PDFを集約するため、OCRやテンプレ設計は不要。**月次バッチで生成** 可能。

---

### 相談②: 既存振込ツールの改修 vs 新規作成、どちらが安全か

**結論**: **新規作成を推奨**（ただし既存ツールのコード参考は必須）

**根拠**:
- WAMは口座が別系統（GMOあおぞら vs 既存PayPay）
- 出力CSVフォーマットが銀行依存（GMOあおぞらフォーマット仕様書: https://gmo-aozora.com/support/paymentguide.html ）
- 既存ツール改修は「**動いている通常振込処理を壊すリスク**」が伴う
- 新規ツールを別モジュールとして作り、**処理本体（BQクエリ）は共通関数で共有** するのが安全

**推奨アプローチ**:
```
既存: bank_transfer_paypay.py (仮称)
  ├─ 共通: fetch_payments_from_bq(member_id, month)  ← 共有関数
  └─ 固有: generate_paypay_csv(payments)

新規: bank_transfer_gmo_aozora.py
  ├─ 共通: fetch_wam_payments_from_bq(member_id, month)  ← 共有関数を流用
  └─ 固有: generate_gmo_aozora_csv(wam_payments)
```

**注意**: **既存の「PayPay銀行一括振込ツール」が本リポジトリ内なのか外部GASなのか、現時点では未把握** です（REQ文書では外部システムと読めます）。本リポジトリ内であればコードを参考にできるので、**Q-C2** でご確認をお願いします。

---

### 相談③: 月別確認ツールは独立ツール vs 既存シートへの集計タブ追加、どちらが開発コスト低いか

**結論**: **既存 `pay-dashboard` への新ページ追加** を推奨（独立ツールとしない）

**根拠**:
- 既存ダッシュボード（`pay-dashboard`）に既に「月別報酬サマリー」タブがあり、`v_monthly_compensation` VIEW で月別・メンバー別の報酬・源泉を算出済み
- WAM版はこのVIEWに **WAMフィルタを追加しただけ** の新VIEW `v_wam_monthly_summary` を作れば、既存のStreamlit UI がほぼそのまま流用できる
- 開発コスト: **新VIEW定義 + 1ページ追加** で約1〜2日（既存パターンの踏襲）

**推奨アプローチ**:
```
dashboard/pages/wam_monthly.py (新規)
  ├─ タイトル: WAM月別報酬・源泉徴収確認
  ├─ フィルタ: 年月 / WAM対象PJ / メンバー
  ├─ 表示: v_wam_monthly_summary からロード
  └─ エクスポート: CSV / Excel
```

**独立ツールが不要な理由**:
- 独立ツールは「認証」「BQ接続」「デプロイ」を全て新規に構築する必要があり、運用負荷が大きい
- 既存ダッシュボードに **roleベースのアクセス制御** （admin/checker/viewer）が既にあるため、WAM担当のみにページを公開することも容易

---

### 相談④: 支払調書ツールへの連携、既存構成を踏まえた実現方法のアドバイス

**結論**: **Want扱いで後回し可** （Phase 3 扱い、Must要件①〜③の実装完了後）

**根拠**:
- 支払調書は年次処理（年1回）なので、初回WAM期間（2026-04-01〜2027-03-31）の完了時点（2027年4月頃）までに実装すればよい
- 本リポジトリ内の `withholding_targets` テーブル（17件の源泉対象マスタ）が既に存在し、`v_monthly_compensation` 経由で **源泉徴収額の計算は既に実装済み**
- したがって、支払調書ツール（外部？）とのインターフェースは「**BQからCSV/JSONでエクスポート**」する形で十分と想定

**補足事項**:
- **Q-C1** で「支払調書作成ツール」がどこにあるか、現物確認をお願いします
- もし本リポジトリ内であれば、既存の源泉計算を流用した改修ができます
- 外部（GAS/Excel）であれば、本ツールはBQからのデータ抽出機能のみ提供し、支払調書ツール側で取り込む形が自然です
