## A. 業務報告シート側の前提確認（5項目）

### Q-A1. 業務報告シートの実体

**質問**: ミヤヤさんのREQドキュメントで言及される「業務報告シート」は、既存 `monthly-pay-tax` システムが既に収集している約190件のシートと**同じもの**か？ それとも別マスタか？

- **想定回答者**: ミヤヤ / ヒロス
- **影響**: 既存インフラの流用可否が確定する
- **調査結果**: 配布リストの「業務報告シートURL」列 = 既存 `gyomu_reports.source_url` と同一の可能性が高い（要確認）

### Q-A2. 業務報告シートの1行の粒度

**質問**: 業務報告シート 1行は何の単位か？（1日単位 / 1タスク単位 / 1活動単位）

- **想定回答者**: ミヤヤ / ヒデス
- **影響**: WAM判別の粒度が決まる（行単位 vs 按分）

### Q-A3. 1行が WAM/非WAM に跨るケースの有無

**質問**: 1件の業務が「一部WAM・一部非WAM」に跨ることはあり得るか？（例: 1日の活動で午前が通常業務、午後が出張タダスクWAM対象）

- **想定回答者**: ミヤヤ / ヒロス
- **影響**: 跨ぐ場合は `activity_category` 単一値による判別が破綻し、按分フィールドまたは行分割入力ルールが必要

### Q-A4. 活動分類プルダウンのWAM対応

**質問**: 業務報告シートの「活動分類」プルダウンに `出張タダスク（WAM対象）` 等の WAM 専用カテゴリを追加することで合意取れるか？

- **想定回答者**: ミヤヤ / タダメン全員（入力運用変更）
- **影響**: Phase 1a の実装方式確定

### Q-A5. 判定方式

**質問**: WAM対象かの判定は「活動分類プルダウンで自動判定」で完結可能か？ それとも追加の人間判断（例: WAM責任者レビュー）が必要か？

- **想定回答者**: ヒデス / ヒロス
- **影響**: 自動判定で済めば VIEW のみ、人間判断が必要なら承認ワークフロー用DMLテーブルが必要

## B. 立替金シート側の前提確認（4項目）

### Q-B1. hojo_reports と立替金シートの関係

**質問**: 現在 `hojo_reports` テーブル（月次集計）が収集している立替額と、各タダメンの **立替金等入力シート** の合計は**一致するか**？ 二重管理なのか、どちらかが正でどちらかが廃止予定なのか？

- **想定回答者**: ヒロス / ゆり
- **影響**: 既存 `hojo_reports` を維持するか廃止するかの方針確定

### Q-B2. 仮払い機能の意味論

**質問**: 立替金シートの「仮払金額」列はWAMフローでどう扱うか？ WAMの会計ルール上、仮払いは許可されるか？

- **想定回答者**: ヒデス / ヒロス
- **影響**: 仮払いが禁止なら該当列は無視、許可ならWAM専用の仮払い処理フローが必要

### Q-B3. 対象PJ プルダウンのWAM対応

**質問**: 立替金シートの「対象PJ」プルダウンに `WAM-出張タダスクPJ` 等の WAM専用PJ名を追加することで合意取れるか？ 配布ツール経由で202件に一括反映可能か？

- **想定回答者**: ゆり / ミヤヤ
- **影響**: Phase 1b の実装方式確定。ゆりさんの配布ツールを活用できれば運用コストゼロで展開可能

### Q-B4. 未配布者の扱い

**質問**: 配布リストでは202件すべて配布済みだが、ユーザー認識では「全員ではない」。**今後新規に立替金シートが必要なメンバー** の発生時、配布ツールで自動対応される認識で合っているか？

- **想定回答者**: ゆり / ミヤヤ
- **影響**: collector の動的シート発見ロジックの要否が決まる
