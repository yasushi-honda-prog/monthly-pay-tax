## ⚠️ 重要: 本ドキュメントは ADR_0005 適用後の最新版

ADR_0005 により「処理・加工は GCP 取得側（pay-collector / BQ / pay-dashboard）で完結させ、既存ツール・シートには触らない」原則を採用しました。この結果、以下の5項目は **GCP側で自己解決可能** と判断し、ステークホルダーへの質問対象から外しました:

- **Q-A4**: 活動分類プルダウンWAM対応 → GCP側のルール分類で吸収
- **Q-B3**: 対象PJプルダウンWAM対応 → GCP側のルール分類で吸収

- **Q-E2**: スマートチップURL挙動 → Sheets API v4 `chipRuns` で吸収
- **Q-E4**: 年度追加ツールWAM対応 → collector の未発行シート吸収

各項目の詳細は該当セクションに `✅ GCP側で吸収` のマークを付けて残しています（実装方針の記録として）。

- 各質問には **想定回答者** と **回答が与える影響** を明記

## A. 業務報告シート側の前提確認（5項目）

### Q-A1. 業務報告シートの実体

**質問**: ミヤヤさんのREQドキュメントで言及される「業務報告シート」は、既存 `monthly-pay-tax` システムが既に収集している約190件のシートと**同じもの**か？ それとも別マスタか？

- **想定回答者**: ミヤヤ / ヒロス
- **影響**: 既存インフラの流用可否が確定する
- **調査結果**: 配布リストの「業務報告シートURL」列 = 既存 `gyomu_reports.source_url` と同一の可能性が高い（要確認）

### Q-A2. 業務報告シートの1行の粒度

**質問**: 業務報告シート 1行は何の単位か？（1日単位 / 1タスク単位 / 1活動単位）

- **想定回答者**: ミヤヤ / ヒデス
- **影響**: WAM判別の粒度が決まる（行単位 vs 按分）

### Q-A3. 1行が WAM/非WAM に跨るケースの有無

**質問**: 1件の業務が「一部WAM・一部非WAM」に跨ることはあり得るか？（例: 1日の活動で午前が通常業務、午後が出張タダスクWAM対象）

- **想定回答者**: ミヤヤ / ヒロス
- **影響**: 跨ぐ場合は `activity_category` 単一値による判別が破綻し、按分フィールドまたは行分割入力ルールが必要

### Q-A4. 活動分類プルダウンのWAM対応 ✅ **GCP側で吸収**（質問不要）

**質問**: 業務報告シートの「活動分類」プルダウンに `出張タダスク（WAM対象）` 等の WAM 専用カテゴリを追加することで合意取れるか？

- **想定回答者**: ~~ミヤヤ / タダメン全員~~ → 不要
- **影響**: ~~Phase 1a の実装方式確定~~
- **ADR_0005 適用後の実装方針**: 活動分類プルダウンは **変更しない**。pay-collector 側で活動カテゴリ × スポンサー × 説明欄テキストをルール分類（必要ならLLM分類）し、BQ 側の新規 `v_wam_monthly_summary` VIEW で自動判定する。タダメンの入力運用は一切変更なし。

### Q-A5. 判定方式

**質問**: WAM対象かの判定は「活動分類プルダウンで自動判定」で完結可能か？ それとも追加の人間判断（例: WAM責任者レビュー）が必要か？

- **想定回答者**: ヒデス / ヒロス
- **影響**: 自動判定で済めば VIEW のみ、人間判断が必要なら承認ワークフロー用DMLテーブルが必要

## B. 立替金シート側の前提確認（4項目）

- **想定回答者**: ヒロス / ゆり
- **影響**: 既存 `hojo_reports` を維持するか廃止するかの方針確定

### Q-B2. 仮払い機能の意味論

**質問**: 立替金シートの「仮払金額」列はWAMフローでどう扱うか？ WAMの会計ルール上、仮払いは許可されるか？

- **想定回答者**: ヒデス / ヒロス
- **影響**: 仮払いが禁止なら該当列は無視、許可ならWAM専用の仮払い処理フローが必要

### Q-B3. 対象PJ プルダウンのWAM対応 ✅ **GCP側で吸収**（質問不要）

**質問**: 立替金シートの「対象PJ」プルダウンに `WAM-出張タダスクPJ` 等の WAM専用PJ名を追加することで合意取れるか？ 配布ツール経由で202件に一括反映可能か？

- **想定回答者**: ~~ゆり / ミヤヤ~~ → 不要
- **影響**: ~~Phase 1b の実装方式確定~~
- **ADR_0005 適用後の実装方針**: 「対象PJ」プルダウンは **変更しない**。既存の `ケアプーPJ / 神奈川県PJ / 経産省PJ / その他` の値 + 説明欄テキストをルール分類し、BQ 側の `wam_target_projects` マスタテーブルで WAM 判定を明示管理する。ゆりさんの配布ツール・既存202件の立替金シートは一切変更なし。

### Q-B4. 未配布者の扱い

**質問**: 配布リストでは202件すべて配布済みだが、ユーザー認識では「全員ではない」。**今後新規に立替金シートが必要なメンバー** の発生時、配布ツールで自動対応される認識で合っているか？

## E. ゆりさん個別相談事項（ADR_0005 適用後: 実質1項目 + GCP吸収3項目）

Phase 1b の実装成否はゆりさんの配布ツールとの連携に大きく依存するため、独立セクションで整理。ADR_0005 により、改修依頼系の3項目（Q-E1, Q-E2, Q-E4）は GCP側で自己解決する方針となり、残る確認項目は Q-E3（運用継続確認）のみ。

### Q-E1. 配布リストへの「立替金シートURL」列追加 ✅ **GCP側で吸収**（質問不要）

**質問**: 現在の配布リストは「業務報告シートURL」＋「既存リンクのセル番地」の組み合わせで立替金シートを参照している（セル位置が K3/L3/K6/K7/K8/K12/L4/L13 等メンバーごとに異なる）。**配布リストに「立替金シートURL」列を追加する** 運用改善は可能か？

- **想定回答者**: ~~ゆり~~ → 不要
- **影響**: ~~pay-collector の実装が大幅にシンプル化される~~
- **ADR_0005 適用後の実装方針**: 配布リストは **変更しない**。pay-collector が配布リストのセル番地文字列（K3/L3/K6等）を動的に解釈し、業務報告シートの該当セルから立替金シートURLを取得する。ゆりさんの配布ツールは現状維持。

### Q-E2. スマートチップ化された参照の扱い ✅ **GCP側で吸収**（質問不要）

**質問**: 配布作業範囲記述書に「スマートチップ化されている場合、検出精度が低下する」との記述がある。Sheets API 経由で業務報告シートのK/L列からURLを取得する場合、スマートチップはどう見えるか？

- **想定回答者**: ~~ゆり~~ → 不要
- **影響**: ~~collector の URL 抽出ロジックの信頼性~~
- **ADR_0005 適用後の実装方針**: Sheets API v4 の `sheets.data.rowData.values.chipRuns` フィールドから smart chip URL を取得可能。`hyperlink` / `textFormatRuns` / `chipRuns` を優先順位で試行し、いずれかから URL を抽出する実装で対応。

### Q-E3. 配布ツールとpay-collectorの責任分界点 ⭕ **残す**（運用継続確認）

**質問**: 立替金シートのデータ収集について、pay-collector がシートを読むだけで完結する方針で進める。この方針で **ゆりさんの配布ツール運用を変更したり止めたりする必要はない** ことの最終確認。

- **想定回答者**: ゆり
- **影響**: ゆりさんの運用負荷ゼロの確認、pay-collector の実装スコープ確定
- **ADR_0005 適用後の立ち位置**: この項目のみ確認が必要（ゆりさんの運用継続の合意取得）

### Q-E4. 年度追加ツールのWAM対応 ✅ **GCP側で吸収**（質問不要）

**質問**: `管理用_立替シート年度追加用` ツールは、令和8年度分のWAM対象カテゴリ追加や、WAM用の新シート発行をサポート可能か？

- **想定回答者**: ~~ゆり~~ → 不要
- **影響**: ~~新年度シートのWAM対応の運用コスト~~
- **ADR_0005 適用後の実装方針**: 年度追加ツールは **変更しない**。新年度（R8）シートが未発行の期間は pay-collector 側でスキップ／後日再取得する実装で吸収。WAM対応のカテゴリ追加も不要（GCP側のルール分類で対応）。
