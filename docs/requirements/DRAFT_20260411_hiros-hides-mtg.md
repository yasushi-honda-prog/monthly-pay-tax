### 2. Q-B1, Q-B2: 立替金・仮払いの会計ルール（10分、主にヒデスさん）

#### Q-B1: hojo_reports と立替金シートの関係 （ヒロス/ゆり）

**質問**: 現在 `hojo_reports` テーブル（月次集計）が収集している立替額と、各タダメンの **立替金等入力シート** の明細合計は**一致しますか**？

**確認したい点**:
- 二重管理なのか、片方が正でもう片方が廃止予定なのか
- 不一致の場合、どちらを「正」として扱うべきか

**選択肢**:
- (A) ✅ **一致する**（同じデータを2つの場所に入力している → 1本化が望ましい）
- (B) **hojo_reports が正**（立替金シートは個人メモ扱い）
- (C) **立替金シートが正**（hojo_reports は概算集計）
- (D) **両方独立**（目的が異なり、並行維持する）

**技術への影響**:
- (A) → 新規 `reimbursement_items` 追加 + `hojo_reports` 廃止提案
- (B) → `hojo_reports` 維持、WAM 対応は別系統で追加
- (C) → `hojo_reports` を `reimbursement_items` 集計で置き換え
- (D) → 両方維持（WAM は `reimbursement_items` のみで対応）

**推奨（ヤスス的意見）**: (C) または (D) — 立替金シート（明細）の方が粒度が細かく、WAM会計監査に耐える

---

#### Q-B2: 仮払い機能の意味論 （ヒデス）

**質問**: 立替金シートの「**仮払金額**」列は、WAMフローでどう扱いますか？ WAM の会計ルール上、仮払いは許可されていますか？

**WAM 事務処理マニュアル確認**:
- 4/9 MTGメモより: 「領収書の宛名は**団体の正式名称**」「**出金伝票は証拠書類として認められない**」
- これらの記述から、**「一時的な仮払い」は許可されるが、最終的に正式領収書で精算必須** と推察

**選択肢**:
- (A) ✅ **WAM では仮払い列は無視**（全て精算後の金額で記録する運用）
- (B) **仮払い列は維持するが、WAM 対象には使わない**（通常PJでのみ使用）
- (C) **仮払い専用のWAM処理フローが必要**（未精算状態での会計計上）
- (D) **仮払い自体をWAMでは禁止する**（入力ルール改訂）

**技術への影響**:
- (A)(D) → 仮払金額列は実在しない（H:Iマージセル調査で判明、#70で修正済み）→ WAM集計で考慮不要
- (B)(C) → 仮払い状態の別管理フローが必要（複雑化）

**推奨（ヤスス的意見）**: (A) — 最もシンプル、WAM 会計リスク最小

### 4. Q-A5: WAM判定方式（5分、ヒデス/ヒロス両方）

#### Q-A5: 判定方式 （ヒデス/ヒロス）

**質問**: 業務報告シート・立替金シートで「WAM対象かどうか」の判定は、**どこまで自動化しますか**？

**選択肢**:
- (A) ✅ **完全自動**（活動分類プルダウン or 対象PJプルダウン の値だけで機械的に判別）
- (B) **半自動 + 人間レビュー**（自動判定結果をWAM責任者が月次でレビュー承認）
- (C) **完全人間判断**（自動判別せず、全件WAM責任者が手動タグ付け）

**技術への影響**:
- (A) → 新規 VIEW (`v_wam_*`) のみで対応可能、ダッシュボード側で表示
- (B) → 承認ワークフロー用DMLテーブル (`wam_review_logs`) が必要、ダッシュボードに承認UIを追加
- (C) → 手動入力フォーム + DMLテーブル必要、運用負荷大

**推奨（ヤスス的意見）**: (A) — ゆりさんの配布ツールが既に統一されているため、プルダウン値の追加だけで完結する

**補足質問**: WAM対象判定の**監査ログ** は必要ですか？（後から「いつ・誰が・何を判定したか」を追跡できる必要があるか）
- 必要 → (B) の承認ワークフローが実質必須
- 不要 → (A) で十分
