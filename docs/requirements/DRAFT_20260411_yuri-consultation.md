お疲れ様です、ヤススです。

令和8年度のWAM助成金事業（2026-04-01〜2027-03-31）への対応を検討しており、**ゆりさんが設計された立替金等入力シート配布ツール** と pay-collector（Cloud Run）の連携設計を進めています。

結論から言うと、**ゆりさんの配布ツール・GASロジックには一切手を入れません**。技術的にGCP側（pay-collector）で吸収できる部分はこちらで対応する方針としました（ADR_0005 として記録済み）。

## 📋 背景

- WAM助成事業では、立替金シートの明細を pay-collector 経由で BigQuery に収集して WAM / 非WAM を分離集計します
- Playwright で配布ツール一式を技術調査させていただき、202件の配布状況・ヘッダー構造・GAS設計を把握済みです（勝手に拝見してすみません 🙇）
- 調査の結果、**配布ツール側は現状のまま、pay-collector 側でシートを読むだけ** で WAM 対応が完結する見通しです

## 🎯 ご確認いただきたい2点

### 確認1（Q-E3）: 配布ツールの運用を変更しないことの最終確認

立替金シートのデータ収集について、以下の方針で進めます:

```
現行の運用（変更なし）:
  ゆりさん作GAS（配布ツール）
    → 立替金シートを配布・維持管理
    → 業務報告シートのセルに URL リンク貼付（K3/L3/K6/K7/K8/K12/L4/L13 等）

追加する処理（ヤスス側で実装）:
  pay-collector（Cloud Run / Python）
    → 業務報告シートの該当セルから立替金シートURLを動的に取得
    → 立替金シートの明細を BigQuery に収集
```

**確認ポイント**: この方針で **ゆりさんの配布ツール運用を止めたり、改修したりする必要は一切発生しません**。pay-collector が「読むだけ」で完結します。

**選択肢**:
- (A) ✅ OK、この方針で進めてください
- (B) 懸念あり（詳細教えてください、例: pay-collector がシートを読むタイミング・頻度など）
- (C) 配布ツール側から統合的に管理したい（GAS拡張案）

→ **ゆりさんのご意見**:

※ ヤスス的には(A)推奨です。ゆりさんの運用負荷をゼロにし、pay-collector 側で既存の190件巡回パターン（毎朝6時バッチ）と同じ方式で収集するため、責任分界が明確になります。

---

### 確認2（Q-B4）: 新規メンバー発生時の立替金シート発行フロー

配布リストでは202件すべて「配布済み」ですが、ユーザー認識では「全員配布されているわけではない」とのことで、少し混乱しています 😅

今後、**新規メンバー追加時** の立替金シート発行は、どのようなフローで行われますか？

**選択肢**:
- (A) ✅ 配布ツールが自動で新規メンバーを検出して発行する
- (B) 手動で発行する（どのタイミング・誰が操作するか教えてください）
- (C) 「全員ではない」認識は誤りで、202件が現時点の全員。新規発生時は都度ゆりさんが手動発行

→ **ゆりさんのご意見**:

※ この回答により、pay-collector 側で「新しいシートを動的に検出するロジック」が必要かどうかが決まります。自動検出（A）なら collector 側の変更は最小、手動発行（B/C）なら collector 側で定期的に配布リストを監視するロジックを実装します。
