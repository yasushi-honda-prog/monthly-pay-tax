# ADR 0005: WAM対応の処理・加工はGCP取得側で完結する

## ステータス
採用 (2026-04-11)

## 背景
WAM助成事業対応（要件①〜④, Issue #54〜#58）のPhase 0 で、ゆりさん・ミヤヤさん・ヒロス/ヒデスの3系統に18項目のステークホルダー質問を整理した。初版の下書きでは以下のような「既存ツール・シートの改修依頼」が含まれていた:

- 配布リストに「立替金シートURL」列を追加してください（Q-E1）
- スマートチップ化URLの挙動を教えてください（Q-E2）
- 年度追加ツールのWAM対応を追加してください（Q-E4）
- 対象PJ プルダウンに `WAM-*` 値を追加してください（Q-B3）
- 活動分類プルダウンに WAM専用カテゴリを追加してください（Q-A4）
- 既存PayPay振込ツールを改修 vs 新規作成、どちらがよいか（相談②）

これらは本質的に「**既存の運用・ツール・シートに触る**」提案であり、以下の問題がある:

1. **ブラスト半径が拡大する**: 動いている運用（202件の立替金シート・190件の業務報告シート・ゆりさんのGAS配布ツール）に変更を加えると、既存機能の回帰リスクが発生する
2. **ステークホルダーの作業負荷が増える**: ゆりさんに配布ツール改修を依頼すると、運用継続の責任がゆりさんに残り続ける
3. **依頼の合意形成に時間がかかる**: 改修可否を確認・決裁する往復でPhase 0 の完了が遅れる
4. **技術的に自己解決可能**: スプレッドシート側のフォーマット揺れやメンバー固有のセル番地差異は、Sheets API v4 の柔軟性（`chipRuns`/動的range/rowData）と pay-collector（Python/Cloud Run）の処理能力で吸収できる

## 検討した選択肢

| 案 | 評価 | 不採用理由 |
|---|---|---|
| **既存ツール改修依頼案**（当初の18項目質問） | ❌ | 既存運用のブラスト半径拡大、ステークホルダー作業負荷増、Phase 0 長期化 |
| **完全新規基盤構築案**（pay-collector とは独立したWAM専用インフラ） | ❌ | 開発コスト膨大、認証・デプロイ・運用を新規構築する必要あり |
| **GCP取得側完結案**（既存インフラに新規テーブル/処理のみ追加） | ⭕ | **採用**: ブラスト半径ゼロ、ステークホルダー負担ゼロ、既存パターン流用で開発コスト最小 |

## 決定
**WAM対応における処理・加工は、すべて pay-collector（Cloud Run / Python）およびBQ層の新規追加で完結させる。既存のシート・GASツール・データ構造には一切変更を加えない。**

### 具体的な適用例

| 当初の改修依頼 | GCP側での実装方針 |
|----------------|-------------------|
| 配布リストにURL列追加（Q-E1） | pay-collector で配布リストのセル番地文字列を動的に解釈し、業務報告シートの該当セルから参照URLを取得 |
| スマートチップ化URL（Q-E2） | Sheets API v4 の `sheets.data.rowData.values.chipRuns` から smart chip URL を抽出（プレーンテキストとは別フィールド） |
| 年度追加ツールのWAM対応（Q-E4） | 新年度シートが未発行の場合は collector 側でスキップ／後日再取得、収集ロジックはフォーマット揺れを吸収 |
| 対象PJ プルダウンにWAM値追加（Q-B3） | 既存の「その他」値 + 説明欄テキストをルールベース/LLM分類で WAM判定、BQ側の `wam_target_projects` マスタで明示管理 |
| 活動分類プルダウンにWAM値追加（Q-A4） | 業務カテゴリ×スポンサー×説明欄の組み合わせで WAM判定、`v_wam_monthly_summary` VIEW で自動分類 |
| PayPay振込ツール改修 vs 新規（相談②） | 新規モジュール `bank_transfer_gmo_aozora.py` を追加、既存 `bank_transfer_paypay.py` は無変更（もし本リポ内であれば） |

### ブラスト半径

```
触らない（既存維持）:
  - コード.js（GAS配布ツール）
  - 立替金等入力シート（202件）
  - 業務報告シート（190件）
  - gyomu_reports / hojo_reports / members テーブル
  - 既存ダッシュボード4タブ
  - 既存VIEW (v_gyomu_enriched, v_hojo_enriched, v_monthly_compensation)

新規追加のみ:
  - reimbursement_items テーブル（立替金明細）
  - wam_target_projects テーブル（WAM判定マスタ）
  - v_wam_monthly_summary VIEW（WAM月別集計）
  - pay-collector に WAM判定・明細収集ステップ追加
  - pay-dashboard に WAM月別確認ページ追加
```

## 期待される効果

1. **Phase 0 の期間短縮**: 18項目質問のうち、GCP側で自己解決できる5項目（Q-A4, Q-B3, Q-E1, Q-E2, Q-E4）を削除、さらに技術決裁で自己解決できる3項目（ミヤヤさん相談①②③）を「決定事項の共有」に変換。ステークホルダー確認が必要な項目は **13項目** に削減
2. **既存運用の安定性確保**: 動作中の202件立替金シートと190件業務報告シートに一切影響なし
3. **ゆりさんの作業負荷ゼロ**: 配布ツール改修依頼を撤回、既存GAS運用は現状維持
4. **テスト範囲の最小化**: 既存テーブル・VIEW・ダッシュボードはスモークテスト不要、新規追加部分のみにテストを集中

## 適用範囲

### 原則が適用される場面
- WAM対応における立替金シート・業務報告シートのデータ収集
- WAM判定ロジック（対象PJ・活動分類の分類）
- WAMフォーマット揺れへの対応
- 既存PDFの集約（要件①）
- 振込CSVの出力（要件②）
- 月別確認ツール（要件③）

### 原則が適用されない場面（例外）
- **会計ルール判断**: 仮払い機能の扱い、WAM/非WAM跨ぎ時の按分ルール、フラット報酬体系など、人間（ヒデスさん/ヒロスさん）の決裁が本質的に必要な事項
- **運用ルール**: WAM判定に人間レビューが必要か（Q-A5）、新規メンバー発生時の発行フロー（Q-B4）など、ビジネスプロセスの決定
- **外部システム連携**: 支払調書ツール・PayPay振込ツールが本リポジトリ外にあり、インターフェース仕様の確認が必要な場合（Q-C1, Q-C2）

## 残課題
- **Q-E3（責任分界点）**: 既存GASによる配布ツール運用を変更しないことの最終確認はゆりさんに必要（「pay-collector がシートを読むだけで、ゆりさん側の運用は止めない」ことの説明）
- **WAM判定ロジックの精度**: 対象PJ「その他」+ 説明欄テキスト分類のルールは、実データでの精度検証が必要（Phase 1a の先頭タスク）
- **新規メンバー発生時の自動対応**: pay-collector が新メンバーのシート発行を動的に検出できるか、配布リストの監視ロジックで対応可能かは要実装検証
